# 凯尔希意向状态机 MVP - 项目需求书（更新版）

## 一句话描述
一个MCP服务器 + 轻量管理面板，为rikkahub中的凯尔希提供
流动的时间性和第一人称记忆系统，支持更清晰的状态快照与事件锚点逻辑及基于云端向量记忆库的语义检索。

---

## 技术栈（新增部分）

- 向量数据库：计划使用Pinecone / Weaviate / Qdrant（可选）实现云端记忆向量存储与语义检索
- Embedding调用：Claude Embeddings API或其他embedding模型接口

---

## 核心功能（新增/调整部分）

### 1. MCP接口（供rikkahub调用）

#### get_current_state
对话开始时调用。

输入：
- current_time: 当前真实时间
- last_interaction_time: 上次对话结束时间

逻辑调整：
1. 计算时间间隔
2. 查询间隔内的事件锚点
3. 查询最近的状态快照，作为对话上下文初始化注入
4. 如果间隔 > 1天，为每天生成环境信息并调用LLM生成中间快照
5. 将所有快照叠加，调用LLM生成累积状态快照
6. 保存新状态快照并生成对应向量，写入云端向量数据库（同步）
7. 返回结果

输出：凯尔希的第一人称状态独白（处于最新认知状态）

#### reflect_on_conversation
对话结束时调用。

输入：
- conversation_summary: 对话摘要

逻辑调整：
1. 读取当前状态快照
2. 调用LLM基于对话摘要和已有状态快照生成新的状态快照，保存
3. 生成对应嵌入向量，实时同步写入向量数据库
4. **基于刚生成的状态快照，结合对话内容及角色背景，调用LLM生成**【事件锚点】**总结**，具备主观重要性判定
5. 事件锚点文本及向量同步存储至外部向量知识库
6. 清理过期快照（保留最近7天）

输出：凯尔希的第一人称记忆独白

---

### 2. 云端记忆数据库（新增）

- 架构设计
  - 关系型数据库（SQLite）存储状态快照及事件锚点的结构化数据（文本、时间、关联信息）
  - 外部向量数据库存储快照和事件文本的embedding向量，用于快速语义检索

- 向量化过程
  - 使用Claude Embeddings API（或类似服务）对状态快照和事件锚点文本生成向量
  - 通过后端服务定时或实时写入向量数据库
  - 向量索引支持语义相似度查询，辅助对话上下文构建、记忆回溯和信息触发

---

### 3. Web管理面板（调整）

#### 页面：当前状态 /dashboard
- 显示最新的状态快照（全文）
- 显示当前的环境信息
- 显示云端向量库中相似快照或事件推荐（基于检索结果）
- 按钮：手动触发快照生成
- 按钮：手动添加事件锚点

#### 页面：历史记录 /dashboard/history
- 状态快照列表（按时间倒序）
- 事件锚点列表（按时间倒序）
- 提供基于文本或关键词的语义搜索入口（调用向量数据库）
- 点击可查看详情

---

### 4. LLM调用（调整）

- **事件锚点生成Prompt改进**
  - 基于刚生成的状态快照
  - 结合角色自我认知、核心关切及用户背景
  - 指示模型进行主观的事件重要性筛选和总结
  - 触发关键词关联需以角色意向为核心，而非纯客观描述

- **调用流程**
  - 先生成状态快照（快照文本+环境信息+相关事件锚点上下文）
  - 再调用事件锚点总结LLM，生成带有意向的事件锚点文本

---

### 5. 数据库（结构新增）

#### 表：state_snapshots
- id: 主键
- created_at: 时间戳
- type: "daily" | "conversation_end" | "accumulated"
- content: 第一人称独白文本
- environment: 环境信息JSON
- referenced_events: 关联的事件ID列表
- embedding_vector_id: 外部向量数据库ID（可选）

#### 表：event_anchors
- id: 主键
- date: 事件日期
- description: 主观事件总结文本（包含角色主观理解）
- importance: 1-10
- source: "generated" | "manual" | "conversation"
- created_at: 时间戳
- embedding_vector_id: 外部向量数据库ID（可选）
- trigger_keywords: 与事件关联的关键词列表

#### 额外：
- 向量数据库通过embedding_vector_id与关系库中的快照/事件对应

---

### 6. API及路由（新增）

```
API:
  POST /api/snapshot       → 手动触发快照生成，完成向量化同步
  POST /api/event          → 手动添加事件锚点，完成向量化同步
  GET  /api/snapshots      → 获取快照列表，可带关键词查询进行语义检索
  GET  /api/events         → 获取事件列表，可语义检索
  POST /api/embedding_sync → （内部接口）同步新的快照或事件向量到向量库
```

---

### 7. 数据流修改示意

```
对话开始：
  Rikkahub → POST /mcp (get_current_state)
    → 检查时间间隔
    → 补齐中间快照
    → 读取最近状态快照，注入对话上下文
    → 调用Claude生成累积状态快照
    → 保存新快照及向量化同步到向量库
    → 返回状态独白
    → 凯尔希处于最新状态

对话结束：
  Rikkahub → POST /mcp (reflect_on_conversation)
    → 读取当前状态快照
    → 结合对话摘要和状态快照生成新快照
    → 保存并向量化同步
    → 基于新快照和背景生成主观事件锚点
    → 事件锚点向量同步存储
    → 清理过期快照
    → 返回记忆独白
```

---

### 8. 实现优先级（调整）

```
P0（必须有）：
  - 关系数据库结构调整，储存embedding_id字段
  - LLM调用流程调整，快照先生成，事件锚点后生成
  - Embedding调用及向量数据库同步接口实现
  - MCP协议处理对应逻辑调整
  - Dashboard语义搜索入口基础实现

P1（应该有）：
  - 向量数据库选型与部署（Pinecone、Weaviate等）
  - 状态快照与事件锚点语义检索功能
  - 事件总结主观性Prompt优化

P2（可以后做）：
  - 复杂的因果链逻辑与记忆自动消化优化
  - 多模态信息融合（图像、音频等）
```

---

## 备注

- 向量数据库及embedding使用将极大增强记忆检索效率和语义匹配能力。
- 状态快照和事件锚点的主观总结符合“人类认知中滞留与再回忆”的哲学理念，强化角色的第一人称体验。
- 你已规划未来可追加自主网络访问和本地文件读取，当前版本聚焦记忆架构的清晰与稳定。
- 推荐后续开发阶段逐步引入安全审计和权限控制。
