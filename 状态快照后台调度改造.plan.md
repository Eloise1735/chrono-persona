# 状态快照后台调度改造 Plan

## 1. 改造目标

将“对话开始时按时间差一次性补生成多个状态快照”的模式，调整为“服务器后台按最小时间单位持续推进快照，对话入口仅做兜底补账”的混合模式，降低以下风险：

- 长间隔后单次请求触发大量 LLM 调用，导致超时、报错或成本暴涨
- 单次请求把过多上下文拼进 prompt，导致上下文膨胀
- 时间推进完全依赖用户触发，请求体验不稳定

同时保留现有系统的两个核心能力：

- 快照仍然要体现“时间流逝中的连续状态变化”
- 事件锚点仍然要覆盖关键客观事件，不能因优化而漏记


## 2. 对现状的判断

### 2.1 `memory_context` 是否在链式堆叠

结论：**不是主要问题，不属于无限链式堆叠。**

当前实现里：

- `memory_context` 来自 `self.memory.search(..., top_k=3)`
- `_format_memories()` 会把每条记忆截断到前 200 字

因此 `memory_context` 本身是有上限的，规模大致稳定。它可能带来“语义噪音”，但不是导致文本量爆炸的主因。

### 2.2 `recent_events` 是否实现偏离预期

结论：**是，当前实现与“只拿最近几条/上一段相关事件”的原意不一致。**

当前 `get_current_state()` 中：

- 用 `get_events_in_range(start_date, end_date)` 一次性取出从上次交互日期到当前日期之间的全部事件
- 这整批事件被格式化成 `events_text`
- 同一份 `events_text` 被传给每一个 checkpoint 的环境生成和快照生成

这会带来三个问题：

1. 事件数量会随间隔时间线性增长，不是“有限最近几条”
2. 早期 checkpoint 会看到后期事件，存在时间泄漏
3. 所有 checkpoint 都反复携带同一大段事件文本，造成无效 token 消耗

### 2.3 真正导致压力变大的因素

主要是以下三项叠加：

1. checkpoint 数量增加，导致 LLM 调用次数增加
2. `recent_events` 按整段时间范围全量注入，而非按 checkpoint 增量注入
3. 每个 checkpoint 都执行“环境生成 + 记忆检索 + 快照生成 + 事件锚点生成”

其中第 2 项是最像“实现错误”的部分，第 1 和第 3 项则是架构选择在长间隔场景下的自然代价。


## 3. 对问题 1 的解决方案

### 3.1 `memory_context`

建议保留“少量检索记忆”的思路，但再加两层约束：

- 仍然限制 `top_k`，默认 2 到 3 条
- 对拼进 prompt 的记忆总字数设置硬上限，超出即截断

可选增强：

- 优先取事件记忆，再取快照记忆
- 对同一天、同主题的检索结果做去重，减少重复语义

### 3.2 `recent_events`

必须改成“按 checkpoint 的时间视角注入”：

- 不再对整个 `last -> now` 区间一次性全取
- 每个 checkpoint 只看到在该 checkpoint 之前已经发生/已经被系统知道的事件
- 每次只注入有限数量，例如最近 3 到 5 条
- 超过上限时先压缩成短摘要再注入

推荐优先级：

1. 先修正时间泄漏
2. 再做条数限制
3. 再做摘要压缩

### 3.3 推荐的数据口径

短期可用口径：

- 以 `created_at <= checkpoint_time` 作为“此时系统已知事件”的判断

中期更优口径：

- 给事件补一个真正的 `occurred_at` 或 `source_time` 字段
- 快照推进时按 `occurred_at <= checkpoint_time` 判断可见事件


## 4. 对问题 2 的解决方案

### 4.1 不是所有场景都必须取消“每步生成事件锚点”

关键判断：

- **稳定后台运行时**：每次只推进 1 个最小时间单位，这时“每步都生成事件锚点”其实是可接受的
- **只在追赶补账时**：才需要考虑减少中间 checkpoint 的事件锚点生成次数

因此，推荐策略不是简单地“以后都只在最后一步生成事件锚点”，而是：

- 实时调度模式：每次推进 1 格，照常尝试生成事件锚点
- 追赶补账模式：中间步骤先记录事件候选，最后做一次汇总/补锚

### 4.2 如何避免中间事件遗漏

推荐采用“两阶段事件处理”：

#### 阶段 A：checkpoint 级事件候选

每个 checkpoint 都保留一份轻量的“事件候选”或“环境事实摘要”，来源可以是：

- 当前环境文本
- 当前快照与上一快照的差异
- 简化版事件检测 prompt

这一步的目标不是生成正式锚点，而是确保“这一格里发生过什么”不会丢。

#### 阶段 B：锚点正式化

在以下时机之一，把若干候选合并成正式事件锚点：

- 单步实时推进完成后，立即生成正式锚点
- 系统追赶补账时，对一个小窗口内的候选做统一归并
- 对话开始前发现有未正式化候选时，补做一次 reconciliation

这样做可以避免两种极端：

- 每一步都完整跑正式事件抽取，成本太高
- 只在最后一步看最终快照，导致中间事件消失

### 4.3 为什么不能只依赖最终快照重新生成事件锚点

因为最终快照更像“当前时刻的主观状态”，它不保证保留完整事件时间线。中间发生过但已过去的客观事件，可能只留下情绪余波，事实骨架会丢失。

所以必须保留“中间步的事实痕迹”。最轻量的方式不是每步都产正式锚点，而是每步都保留可回溯的候选事实。

### 4.4 “没有 `recent_events` 就不实时”的问题怎么解决

这个担心是对的，但解决方式不是把所有最近事件都塞进去，而是把 `recent_events` 改成：

- 只看当前 checkpoint 前可见的事件
- 只保留最近几条
- 超过条数后先摘要
- 必要时区分“硬事件”和“软背景”

建议拆成两个注入块：

- `recent_event_digest`：最近 3 到 5 条明确事件
- `continuity_digest`：上一环境与上一快照延续出的背景线索

这样快照依旧有实时性，但不会因整段时间线全量灌入而失控。


## 5. 推荐的目标架构

采用“后台调度 + 前台兜底 + 候选事件缓冲”的混合架构。

### 5.1 后台调度器

服务器启动后创建一个后台协程：

- 固定频率轮询，例如每 10 到 15 分钟检查一次
- 如果 `now - latest_snapshot_time >= min_time_unit`，只推进 1 个 checkpoint
- 一次最多推进 1 格，防止单次后台任务过重

### 5.2 前台兜底补账

`get_current_state()` 仍然保留，但职责收缩为：

- 正常情况下直接返回当前最新状态
- 若后台因为重启/休眠落后，则做有限补账
- 单次最多补 1 到 3 格，剩余部分交给后台继续追赶

### 5.3 推进逻辑统一

需要把当前 `get_current_state()` 里的单步推进逻辑抽成公共方法，例如：

- `advance_one_checkpoint()`
- `advance_until(target_time, max_steps=...)`

后台调度器和前台兜底都调用同一套推进逻辑，避免行为分叉。

### 5.4 事件处理策略

推荐分模式：

- 实时单步推进：快照 + 正式事件锚点
- 追赶补账：快照 + 轻量候选事件，最后做统一补锚


## 6. 必要的约束与保护

### 6.1 并发保护

必须加锁，避免这些入口同时推进状态：

- 后台调度器
- `/state/current`
- 未来可能新增的手动补账接口

建议：

- 在 `StateMachine` 内维护一个全局 `asyncio.Lock`
- 所有“时间推进”相关入口统一走同一把锁

### 6.2 token / 文本预算

需要对以下输入分别设置硬预算：

- 上一快照
- recent events
- memory context
- environment text

超过预算时的处理顺序：

1. 先裁剪事件列表
2. 再裁剪记忆列表
3. 必要时将历史块摘要化

### 6.3 时间因果一致性

每个 checkpoint 只能看到该时刻之前的内容：

- 事件
- 候选事件
- 已生成快照
- 环境连续性信息

严禁把未来 checkpoint 的事件泄漏给过去 checkpoint。


## 7. 分阶段实施步骤

### Phase 0：观测与防误判

目标：先看清真实膨胀点。

任务：

- 给 `get_current_state()` 打印每个 checkpoint 的输入规模
- 统计每轮 prompt 中：
  - `previous_snapshot` 长度
  - `recent_events` 条数与字符数
  - `memory_context` 条数与字符数
  - 每次总 token
- 确认最常见报错场景是超时、上下文过长还是网关响应异常

验收：

- 能通过日志回答“到底是谁在膨胀”

### Phase 1：修正现有按需补账的逻辑偏差

目标：不改架构，先修掉最明显的不合理实现。

任务：

- 将 `recent_events` 从“整段范围全量注入”改为“按 checkpoint 过滤”
- 加 `recent_events` 的条数上限
- 给 `recent_events` 加字符预算
- 保持 `memory_context` 继续限制 `top_k`
- 给 `memory_context` 增加总字符上限

验收：

- 早期 checkpoint 不再看到未来事件
- 长间隔时单轮 prompt 大小显著下降

### Phase 2：抽出统一推进内核

目标：为后台调度做准备。

任务：

- 将当前循环中的单步逻辑提炼为独立方法
- 将“多步推进”封装为统一接口
- 明确区分：
  - 实时单步推进
  - catch-up 多步推进

验收：

- `get_current_state()` 与未来后台任务共享同一套推进代码

### Phase 3：加入后台调度器

目标：把大部分补账工作从请求链路搬到服务器后台。

任务：

- 在应用启动生命周期中创建后台协程
- 周期性检查是否到达下一个快照时刻
- 每次只推进 1 格
- 服务关闭时优雅取消后台任务

验收：

- 服务稳定运行一段时间后，即使没有用户请求，也会自主生成新快照

### Phase 4：加入 catch-up 模式与事件候选机制

目标：解决重启、宕机、休眠后的追赶问题，同时避免事件遗漏。

任务：

- 定义“追赶补账模式”
- 中间 checkpoint 写入轻量候选事件/事实摘要
- 在追赶结束时做统一事件锚点补全
- 给候选事件加“已正式化”标记，防止重复入库

验收：

- 系统落后多个时间单位后追赶，能保留中间事件线索
- 不会因为只看最终快照而漏掉中间事件

### Phase 5：前台入口收口为兜底模式

目标：把 `get_current_state()` 从“主推进器”降为“兜底器”。

任务：

- 正常情况下只读取最新快照
- 若后台落后，则有限补账
- 返回当前状态时附带调度元信息，便于调试

验收：

- 大多数 `/state/current` 请求不再触发长链路 LLM 补账


## 8. 数据结构建议

### 8.1 最小改动方案

不新增表，先在内存或现有快照 `environment` 字段中保存临时候选。

优点：

- 改动小
- 快速验证思路

缺点：

- 候选事件管理不清晰
- 重启后恢复能力差

### 8.2 推荐方案

新增一张轻量表，例如 `pending_event_candidates`：

- `id`
- `checkpoint_time`
- `snapshot_id`
- `raw_fact_text`
- `source_env_summary`
- `finalized` 是否已正式生成锚点
- `created_at`

优点：

- 可追踪
- 可重试
- 可防重


## 9. 验收标准

改造完成后，应满足以下条件：

- 长时间无人对话后，下一次 `/state/current` 不再触发大批量补生成
- 单个 checkpoint 的 prompt 规模有明确上限
- 过去 checkpoint 不会看到未来事件
- 后台调度和前台兜底不会并发重复生成快照
- 中间 checkpoint 的重要事件不会在 catch-up 时丢失
- 服务重启后仍可继续追赶，不产生大量重复事件锚点


## 10. 风险与回滚

### 10.1 主要风险

- 后台调度与前台请求竞争，产生重复快照
- 事件候选与正式锚点之间的去重不充分
- 部署环境不稳定，后台任务不能持续运行
- 新增后台任务后，日志和故障排查复杂度提升

### 10.2 回滚策略

保留以下开关：

- `snapshot_scheduler_enabled`
- `snapshot_scheduler_interval_sec`
- `snapshot_catchup_max_steps_per_run`
- `snapshot_event_candidate_enabled`

若上线后不稳定：

- 先关闭后台调度器
- 保留 Phase 1 的上下文修正
- 回退到“请求触发补账，但上下文已受控”的模式


## 11. 建议的实施顺序

推荐顺序如下：

1. 先做 Phase 0 和 Phase 1
2. 确认 `recent_events` 修正后，看看报错是否已明显下降
3. 再做 Phase 2 和 Phase 3
4. 最后视追赶场景是否频繁，再决定是否做 Phase 4 的候选事件机制

换句话说：

- **不是一定要一步到位把“后台调度 + 候选事件系统”全部上完**
- **最先值得修的是 `recent_events` 的时间泄漏与无上限注入**


## 12. 给未来 Codex 的执行提示

若后续在 Cursor 中让 Codex 继续推进本计划，建议按以下顺序下达任务：

1. “先不要改架构，先修复 `get_current_state()` 中 `recent_events` 全量注入和时间泄漏问题，并增加日志观测。”
2. “把单步 checkpoint 推进逻辑从 `get_current_state()` 中抽成公共方法，行为保持不变。”
3. “在 FastAPI lifespan 中加入后台 scheduler，但先只做单步推进，不做候选事件表。”
4. “如果 catch-up 时仍担心事件遗漏，再实现 `pending_event_candidates` 机制和补锚流程。”
